Chapter 22 — Multithreading & Multiprocessing

Learn how to run work concurrently in Python using threads and processes. Understand common issues like race conditions and how to avoid them with locks and safe data structures.

1) Introduction

Sometimes your program waits: for network, disk, or user input.

Concurrency lets other work run while you wait.

Threads are lightweight; processes are heavier but avoid many memory-sharing bugs.

2) Theory

2.1 Concurrency vs parallelism

  • Concurrency: tasks overlap in time.
  • Parallelism: tasks run at the same moment on different CPU cores.

2.2 Threads and the GIL (concept)

In CPython, the GIL (Global Interpreter Lock) can limit true parallel execution of Python bytecode. Still, threads can be very effective for I/O-bound work.

2.3 Multiprocessing

Processes have their own Python interpreter and memory. This helps for CPU-bound tasks (more true parallelism).

2.4 Race conditions

A race condition happens when two workers change the same data at the same time and the outcome depends on timing.

⚠ Warning

If threads share memory, you must protect shared variables with locks. Without locks, you may get wrong counts and rare bugs.

3) Syntax

Python
thread example
import threading
import time

def worker(name):
    # Simulate I/O wait
    time.sleep(0.1)
    print("Done:", name)

threads = []
for i in range(3):
    t = threading.Thread(target=worker, args=(f"job-{i}",))
    threads.append(t)
    t.start()

# Wait for all threads
for t in threads:
    t.join()
Python
multiprocessing example
import multiprocessing as mp

def work(x):
    # CPU-bound concept: square
    return x * x

if __name__ == "__main__":
    with mp.Pool(processes=2) as pool:
        results = pool.map(work, [1,2,3,4])
        print(results)

Time: depends on task type • Space: depends on data copies.

Alternative: use concurrent.futures (simple API for threads/process pools).

4) Examples

4.1 Easy Example — thread-safe counter (lock)

Python
lock protects shared value
import threading

counter = 0
counter_lock = threading.Lock()

def inc(n):
    global counter
    for _ in range(n):
        # Lock makes update atomic (concept)
        with counter_lock:
            counter += 1

threads = [
    threading.Thread(target=inc, args=(5000,)),
    threading.Thread(target=inc, args=(5000,)),
]

for t in threads:
    t.start()
for t in threads:
    t.join()

print("counter =", counter)  # Expected: 10000

Output: counter should be exactly 10000.

Time: O(n) increments • Space: O(1) extra.

4.2 Medium Example — race condition demo (no lock)

Why it fails

counter += 1 is not atomic in Python. Two threads can interleave and lose updates.

Python
race condition risk
import threading

counter = 0

def inc(n):
    global counter
    for _ in range(n):
        counter += 1  # not protected

threads = [
    threading.Thread(target=inc, args=(50000,)),
    threading.Thread(target=inc, args=(50000,)),
]

for t in threads:
    t.start()
for t in threads:
    t.join()

print("counter =", counter)

Output: may be less than 100000 (depends on timing).

4.3 Advanced Example — thread pool for I/O tasks

Python
concurrent futures concept
from concurrent.futures import ThreadPoolExecutor
import time

def fetch_sim(url):
    # I/O wait simulation
    time.sleep(0.2)
    return f"data for {url}"

urls = ["a.com", "b.com", "c.com"]

with ThreadPoolExecutor(max_workers=3) as ex:
    results = list(ex.map(fetch_sim, urls))

print(results)

Time: roughly O(total_wait / workers) for I/O simulation.
