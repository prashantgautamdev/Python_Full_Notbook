1) Introduction

Writing code is only half the job. The other half is shipping it so real users can run it.

Deployment is the process of turning your code into something that runs reliably. DevOps adds automation and monitoring so releases are consistent.

2) Theory

2.1 What is deployment?

Deployment means copying your app to a place where users can access it. The place could be a server, a container, a cloud service, or even a static host.

2.2 What is DevOps?

DevOps is a culture + process. It focuses on fast feedback and reliable shipping.

2.3 CI/CD (concept)

  • CI = Continuous Integration: run tests on every change.
  • CD = Continuous Delivery/Deployment: automate releases.

2.4 Environments and dependencies

A big source of bugs is “works on my machine”. DevOps aims to make environments predictable.

⚠ Warning

Never deploy with unpinned dependencies unless you fully understand the risk. Updates can break your app.

3) Syntax

Python
read version from a file (concept)
from pathlib import Path

# Read app version from a text file
# (This keeps one source of truth.)
version = Path("VERSION.txt").read_text(encoding="utf-8").strip()

print("App version:", version)

Time: O(size of file) • Space: O(size of file).

Alternative: store version in a Python variable inside a package.

4.2 Medium Example — “health check” function

Python
simple runtime check
import time

def health_check():
    """Return True if the service seems healthy."""
    start = time.time()

    # Concept: replace with real checks (DB ping, API dependency ping)
    ok = True

    elapsed = time.time() - start
    print(f"Health check took {elapsed:.6f}s")
    return ok

print("Healthy?", health_check())

Output: Prints time and True/False.

4.3 Advanced Example — versioned release idea

Python
generate a release tag (concept)
from datetime import datetime

# In real projects, your CI would create release tags.
# Here we build a tag string.

def make_release_tag(version):
    now = datetime.utcnow().strftime("%Y%m%d")
    return f"v{version}-{now}"

print(make_release_tag("1.4.0"))

Time: O(1) • Space: O(1).

Alternative: use Git commit SHA in tags.

15) Debugging Practice

Case 1: missing environment variables

bug
crash without validation
import os

prod_url = os.environ["PROD_URL"]  # KeyError if not set
print(prod_url)

Fix: use os.environ.get() and print a friendly message.

Case 2: dry-run writes files

bug
writes even in dry-run
# dry_run should not write artifacts
open("manifest.json","w").write("{}")

Fix: guard writes with if not dry_run:.

Case 3: deploy uses wrong version

bug
version mismatch
# supposed to use release tag, but uses local version variable
print("Deploying:", local_version)

Fix: always deploy the version from the manifest/artifact.

16) Common Errors

  • KeyError: missing env var.
  • RuntimeError: stage fails but pipeline continues.
  • PermissionError: writing artifacts where you don’t have rights.
  • Dependency mismatch: unpinned versions.

17) Best Practices

  • Pin dependencies.
  • Test automatically (CI idea).
  • Use staging before production.
  • Keep releases versioned.
  • Build rollback/restore plan.
  • Log enough to debug after deployment.

22) Answer Key (Explain every answer)

Key explanations

  • Deployment: shipping your code so users can run it.
  • DevOps: automating delivery + feedback + reliability.
  • CI: tests/quality checks on each change.
  • CD: automate release after CI passes.
  • Pinning: prevents random dependency updates from breaking your app.
  • Health checks: quick checks that services are responding.
  • Rollback: revert to a stable version if something goes wrong.
