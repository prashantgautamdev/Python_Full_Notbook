1) Introduction

When you change code, you want a history. Git records changes as snapshots.

GitHub stores Git repositories online and supports collaboration.

2) Theory

2.1 Version control

Version control is a system that tracks changes over time. You can go back to earlier versions when something breaks.

2.2 Commits as snapshots

A commit stores a snapshot of your project. It includes a message like “Fix bug in parser”.

2.3 Branches

A branch is a separate line of development. You can work on a feature without breaking the main line.

⚠ Warning

Avoid committing secrets (API keys, passwords). Use .gitignore to hide temporary files.

3) Syntax

Git
basic flow
# initialize repository
git init

# check status
git status

# stage files
git add .

# commit snapshot
git commit -m "First commit"
Git
branch and merge
# create a new branch
git checkout -b feature-login

# after work is done
git add .
git commit -m "Add login feature"

# switch back and merge
git checkout main
git merge feature-login

Time: most operations depend on changed files • Space: repository history storage.

Alternative: use GUI Git clients, but understanding commands helps debugging.

4) Examples

4.1 Easy Example — track changes

Git
status → add → commit
git status
# shows modified files

git add README.md
git commit -m "Update README"

Output: “nothing to commit” after a successful commit.

4.2 Medium Example — push to GitHub

Git
connect remote
git remote add origin https://github.com/USERNAME/REPOSITORY.git

git branch -M main

git push -u origin main

4.3 Advanced Example — pull request concept

A pull request is a request to merge a branch into main. It enables code review.

Flow idea

feature branch → push → open PR → review → merge

7) Dry Run

Dry run
predict git status
Start: repo clean
1) Edit README.md
2) Run git status → shows README.md as modified
3) Run git add README.md
4) Run git status → shows staged changes (not committed yet)
5) Run git commit -m "msg"
6) Run git status → clean (nothing to commit)

Time: depends on git diff size.

15) Debugging Practice

Case 1: nothing to commit

bug
already committed
git status
# says nothing to commit

Fix: make changes first, then stage and commit.

Case 2: conflict on merge

bug
same line changed
CONFLICT (content): Merge conflict in file.txt

Fix: open conflict file, choose correct content, then commit.

Case 3: forgot to add file

bug
not staged
# file is modified but not staged
git commit -m "msg"

Fix: run git add for that file.

16) Common Errors

  • non-fast-forward: remote has new commits.
  • merge conflict: overlapping changes.
  • authentication error: wrong credentials/token.
  • pathspec error: wrong filename when staging.

17) Best Practices

  • Commit often with clear messages.
  • Keep commits focused (one logical change).
  • Use branches for features.
  • Review diffs before committing.
  • Use .gitignore for temporary files and caches.

18) Summary
