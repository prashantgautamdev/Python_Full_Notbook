1) Introduction

Capstone projects show you can build software, not only learn concepts.

In this capstone, you will design a small organizer tool: it reads a tasks file, marks completion, validates entries, and exports a report.

2) Theory

2.1 Problem breakdown

Think in 4 parts:

  • Input: a JSON tasks file (or CSV).
  • Validation: ensure fields are correct.
  • Processing: update statuses + compute summary.
  • Output: export a text/JSON report.

2.2 Design with modules (concept)

  • models.py — data structures
  • io.py — read/write tasks
  • logic.py — validation and report calculations
  • cli.py — command parsing and calling logic

⚠ Warning

Don’t mix file reading with business logic. Keep validation and processing separate so testing becomes easy.

2.3 Testing mindset

Test pure functions first. If your logic is pure, you can test it without files.

3) Syntax

Python
task validation + report (core logic)
from dataclasses import dataclass

@dataclass
class Task:
    id: int
    title: str
    done: bool


def validate_tasks(raw_tasks):
    """Return list[Task] after validation.

    Time complexity: O(n) over tasks.
    Space complexity: O(n) for validated objects.

    Alternative: validate using JSON schema libraries (advanced).
    """
    tasks = []

    for t in raw_tasks:
        # Validate required keys
        if "id" not in t or "title" not in t or "done" not in t:
            raise ValueError("Each task must have id, title, done")

        # Validate types
        if not isinstance(t["id"], int):
            raise TypeError("Task id must be int")
        if not isinstance(t["title"], str) or not t["title"].strip():
            raise ValueError("Task title must be non-empty string")
        if not isinstance(t["done"], bool):
            raise TypeError("Task done must be boolean")

        tasks.append(Task(id=t["id"], title=t["title"], done=t["done"]))

    return tasks


def build_report(tasks):
    """Build a report string.

    Time: O(n) • Space: O(1) extra (report building aside).
    Alternative: return structured dict instead of text.
    """
    total = len(tasks)
    done_count = sum(1 for x in tasks if x.done)
    pending = total - done_count

    lines = []
    lines.append(f"Total tasks: {total}")
    lines.append(f"Done: {done_count}")
    lines.append(f"Pending: {pending}")

    # List pending tasks
    lines.append("Pending titles:")
    for x in tasks:
        if not x.done:
            lines.append(f"- [{x.id}] {x.title}")

    return "\n".join(lines)

Output: validated tasks list and a report string.

Time: O(n) • Space: O(n).

Alternative: keep tasks as dicts (simpler) but dataclass improves clarity.

4.2 Medium Example — update a task status (concept)

Python
mark done by id
def mark_done(tasks, task_id):
    """Mark the task as done.

    Time: O(n) to find a task.
    Space: O(1) extra.

    Alternative: store tasks in a dict by id for O(1) lookup.
    """
    for t in tasks:
        if t.id == task_id:
            t.done = True
            return True
    return False

4.3 Advanced Example — dict-based tasks for faster updates

Python
id → Task mapping
def tasks_to_dict(tasks):
    # build lookup dict
    return {t.id: t for t in tasks}

# Now mark_done can be:
# tasks_map[task_id].done = True

15) Debugging Practice

Case 1: validation passes wrong type

bug
string id accepted
if isinstance(t["id"], (int, str)):  # wrong

Fix: validate strict types and raise clear errors.

Case 2: report counts wrong

bug
done_count computed wrong
done_count = len(tasks)  # wrong

Fix: recompute with sum(1 for x in tasks if x.done).

Case 3: writing files in dry-run

bug
dry_run not respected
if dry_run:
    print("Dry run")
# still writes afterwards

Fix: wrap write logic inside if not dry_run or return early.

16) Common Errors

  • KeyError: missing task fields in raw JSON.
  • TypeError: wrong type for done or id.
  • ValueError: empty title string.
  • JSONDecodeError: corrupted file input.

17) Best Practices

  • Keep logic pure and testable.
  • Validate inputs and raise clear errors.
  • Add dry-run before destructive writes.
  • Write unit tests for validation and summary.
  • Document file formats and CLI usage.
