1) Introduction

Bugs are normal. The difference between beginners and professionals is how fast they find and fix bugs.

In this chapter, you learn a simple testing habit and a structured debugging process.

2) Theory

2.1 Debugging vs testing

  • Debugging: find the cause of a current bug.
  • Testing: prevent future bugs by checking behavior.

2.2 The test idea (concept)

A test checks: for a given input, we expect a specific output. If output differs, the test fails.

2.3 Asserts (quick tests)

assert condition means “this must be true”. If not, Python raises AssertionError.

⚠ Warning

Don’t only test happy paths. Always test edge cases (empty input, wrong types, missing keys).

3) Syntax

Python
assert test
def add(a, b):
    # Adds two numbers
    return a + b

# Test 1
assert add(2, 3) == 5, "2+3 should be 5"

# Test 2
assert add(-1, 1) == 0, "-1+1 should be 0"

print("All tests passed")

Time: depends on function • Space: depends on inputs/outputs.

Alternative: later you will learn unittest or pytest frameworks.

4) Examples

4.1 Easy Example — output prediction

⭐ Remember

Prediction is a test for your brain. It catches mistakes before running code.

Python
predict output
x = 10
if x > 5:
    msg = "big"
else:
    msg = "small"

print(msg)

Output: big (because 10 > 5).

4.2 Medium Example — test edge cases (concept)

Python
edge tests
def safe_divide(a, b):
    # Divide a by b, but handle b=0
    if b == 0:
        return None
    return a / b

assert safe_divide(10, 2) == 5
assert safe_divide(10, 0) is None

print("Division tests ok")

4.3 Advanced Example — testing exceptions (concept)

Python
check error behavior
def must_be_int(x):
    # Converts x to int or raises ValueError
    return int(x)

try:
    must_be_int("abc")
    assert False, "Expected ValueError"
except ValueError:
    # Correct behavior
    print("Got expected ValueError")

Output: Got expected ValueError.

7) Dry Run

Dry run
assert failing
assert add(2, 3) == 6, "Expected 6"

Step:
1) add(2,3) returns 5
2) compare 5 == 6 → false
3) raise AssertionError with message

Program stops at first failing assert.

Time: O(number_of_tests) • Space: small extra.

15) Debugging Practice

Case 1: wrong comparison

bug
> vs >=
def is_adult(age):
    # should be adult if age >= 18
    return age > 18

Fix: use age >= 18. Add a test with age=18.

Case 2: missing return

bug
returns None
def add(a,b):
    result = a+b
    # forgot return

Fix: return result. Tests should catch it.

Case 3: type mismatch

bug
string numbers
nums = ["2","3","4"]
print(sum(nums))  # crash

Fix: convert strings to integers before sum: sum(int(x) for x in nums).

16) Common Errors

  • AssertionError: a behavior didn’t match expected output.
  • NameError: variable/function not defined.
  • TypeError: wrong type passed to operation.
  • ZeroDivisionError: dividing by zero.
  • ValueError: invalid conversion/parsing.

17) Best Practices

  • Write tests for tricky parts and edge cases.
  • After fixing a bug, add a test that would have caught it.
  • Use clear assert messages.
  • Debug with a checklist: reproduce → isolate → inspect → hypothesize → retest.
  • Keep functions small so testing is easy.

22) Answer Key (Explain every answer)

Key explanations

  • Testing: checks that code returns expected results for inputs.
  • Debugging: finds the actual cause of a bug by reproducing and inspecting.
  • assert: verifies assumptions; fails fast with AssertionError.
  • Edge cases: reveal hidden bugs (empty input, wrong types, boundaries).
  • Exceptions testing: use try/except to confirm expected errors occur.
  • Regression: an old bug comes back; tests prevent this.
