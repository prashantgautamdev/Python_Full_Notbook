1) Introduction

Real projects must work for different users and inputs. They also must be maintainable.

This chapter teaches how to build real Python projects using the skills you already learned.

2) Theory

2.1 Project workflow

  • Problem: what should the program do?
  • Inputs: what data comes in?
  • Outputs: what results should it produce?
  • Design: modules/functions/classes.
  • Build: implement small steps.
  • Test: verify with examples and edge cases.
  • Refactor: clean code, add comments.
  • Deploy: package and share (concept).

2.2 CLI vs small web vs automation

  • CLI tool: best for data processing and automation.
  • Small web: best for user interfaces.
  • Automation: best for repetitive tasks.

⚠ Warning

Avoid starting with the “perfect” full project. Start with the smallest working version (MVP).

3) Syntax

Python
mini CLI parser (concept)
import argparse


def main():
    parser = argparse.ArgumentParser(description="Simple word counter")
    parser.add_argument("text", help="Text to analyze")
    args = parser.parse_args()

    # Core logic
    words = args.text.split()
    print("Words:", len(words))


if __name__ == "__main__":
    main()

Output: prints number of words in the input text.

Time: O(n) • Space: O(k) for word list.

Alternative: handle streaming input for very large text.

4) Examples

4.1 Easy Example — file summarizer

Python
read a text file and count lines
def summarize_text(path):
    # Read file and count lines
    # Time O(n lines) • Space O(1) if we stream
    with open(path, "r", encoding="utf-8") as f:
        line_count = 0
        for _ in f:
            line_count += 1

    return {"lines": line_count}

print(summarize_text("notes.txt"))

Output: dictionary with the line count.

4.2 Medium Example — automation concept

Concept
rename files by pattern
1) Scan a folder for files.
2) For each file, extract date from its name.
3) Rename to a standard format.
4) Print a dry-run preview first.

Dry run helps you avoid destroying data.

4.3 Advanced Example — end-to-end mini pipeline

Flow idea

  • Load: save results to a new file/DB.

15) Debugging Practice

Case 1: wrong file encoding

bug
UnicodeDecodeError
with open(path, "r") as f:
    data = f.read()

Fix: use encoding="utf-8".

Case 2: JSON not writable

bug
permission error
with open(out, "w") as f:
    json.dump(data, f)

Fix: validate paths and handle exceptions with friendly messages.

Case 3: logic error in dry-run

bug
writes even in dry-run
if dry_run:
    pass
# then still writes

Fix: return early or wrap write code inside if not dry_run.

16) Common Errors

  • FileNotFoundError: input path wrong.
  • JSONDecodeError: corrupted JSON file.
  • KeyError: missing key in saved data.
  • ValueError: invalid input format.

17) Best Practices

  • Build MVP first.
  • Write small tests early.
  • Stream files for large inputs.
  • Add dry-run for destructive actions.
  • Keep functions small and named clearly.

22) Answer Key (Explain every answer)

Key explanations

  • MVP: minimum viable product that works end-to-end with basic features.
  • Dry-run: preview changes without writing/deleting to avoid damage.
  • Pipeline: structured steps to process data from input to output.
  • ETL: extract data, transform (clean/compute), then load/save results.
  • Edge tests: verify unusual inputs don’t break logic.
  • Refactor: improve code structure without changing behavior.
  • Documentation: helps users and maintainers understand usage.
