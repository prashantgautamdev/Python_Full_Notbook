1) Introduction

Slow code feels like a mystery. In reality, it’s usually one part of the code doing too much work.

Performance optimization is the skill of finding bottlenecks and fixing them carefully.

2) Theory

2.1 Bottlenecks and profiling

Profiling means measuring where time is spent. Without measurement, you may optimize the wrong thing.

2.2 Algorithm first

Changing an algorithm can give huge speedups. Example: linear search (O(n)) vs binary search (O(log n)).

2.3 Reduce work

  • Avoid repeated calculations.
  • Move invariant computations outside loops.
  • Use faster data structures for membership checks.

2.4 Cache and memoization

Caching stores results so future calls reuse them. Memoization is caching inside functions.

⚠ Warning

Caching can increase memory usage. Always consider the time/space trade-off.

3) Syntax

Python
time measurement + caching concept
import time
from functools import lru_cache

@lru_cache(maxsize=None)
def fib(n):
    # Time complexity improves with memoization (concept)
    if n <= 1:
        return n
    return fib(n-1) + fib(n-2)

start = time.perf_counter()
print(fib(35))
end = time.perf_counter()

print("Seconds:", end - start)

Output: Fibonacci number plus elapsed seconds.

Time: reduced drastically with memoization • Space: cache uses memory.

Alternative: dynamic programming table (iterative) can also work efficiently.

4) Examples

4.1 Easy Example — membership: list vs set

Python
choose the right data structure
items_list = [1,2,3,4,5,6,7,8,9]
items_set = set(items_list)

# Concept: list membership is O(n), set membership is ~O(1) average.

target = 7
print(target in items_list)  # True
print(target in items_set)   # True

Output: True / True.

Time: list membership could be O(n); set membership ~O(1).

4.2 Medium Example — caching repeated computation

Python
memoization
from functools import lru_cache

@lru_cache(maxsize=None)
def pow2(x):
    # If called repeatedly with same x, cache saves time.
    return x * x

print(pow2(10))
print(pow2(10))  # second call reused from cache

Output: 100 then 100 again (faster second call concept).

4.3 Advanced Example — avoid work inside loops

Python
move invariant code out
import math

def slow(nums):
    out = []
    for x in nums:
        # recompute constant every iteration (bad)
        out.append(x + math.sqrt(9))
    return out

def fast(nums):
    const = math.sqrt(9)  # compute once
    out = []
    for x in nums:
        out.append(x + const)
    return out

Time: both O(n) but fast version reduces constant work.

7) Dry Run

Dry run
memoization idea
Suppose fib(5) needs fib(4) + fib(3)
Without memoization, fib(3) may be recomputed many times.
With memoization:
1) compute fib(3) once
2) store result
3) reuse stored result next time
So fewer repeated calls happen.

Time: repeated computations removed • Space: cache stores values.

15) Debugging Practice

Case 1: caching stale results

bug
wrong cache key
# cache only by first param, ignoring second

Fix: include all parameters in cache key.

Case 2: optimization removed correctness

bug
moved code changes behavior
# invariant wasn’t actually invariant

Fix: verify assumptions with tests.

Case 3: benchmark compares unfair cases

bug
different inputs
# ran functions with different data

Fix: use the same input data for all comparisons.

16) Common Errors

  • Wrong optimization: changing output while trying to speed up.
  • Memory blow-up: unbounded caches.
  • Benchmark mistakes: timing different workloads.
  • Premature optimization: optimizing before finding bottleneck.

17) Best Practices

  • Measure first, optimize second.
  • Pick the biggest bottleneck.
  • Prefer algorithm/data structure changes over micro tweaks.
  • Use caching only when it saves time significantly.
  • Test correctness after every change.

18) Summary

You learned the optimization mindset: measure → find bottleneck → improve with algorithms/data structures/caching → test → re-measure. You also practiced common techniques like using sets for membership, memoization, and moving loop invariants.

19) Cheat Sheet

Problem Fix Why
Slow membership use set hashing → fast average lookup
Repeated computation cache/memoize avoid recomputation
Expensive inside loop move invariant out compute once
Searching in unsorted data consider sort + binary search O(log n) search

22) Answer Key (Explain every answer)

Key explanations

  • Profiling: shows where time is spent so you optimize the real bottleneck.
  • Big-O: helps compare scaling behavior as input grows.
  • Caching: reuses results to avoid repeated work.
  • Trade-off: caching uses memory for faster time.
  • Invariant: something that does not change across loop iterations.
  • Data structures: choosing sets/dicts can reduce time from O(n) to ~O(1).
